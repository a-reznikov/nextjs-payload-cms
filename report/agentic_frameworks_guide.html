Shared Development Model

Base Spec-Driven Flow

The tested frameworks differ in commands, skills, artifacts, and automation depth, but the core pattern is the same: turn an idea into explicit requirements, break those requirements into executable tasks, implement, and verify the result against the spec.

Tool-specific details live in each framework profile
1

Idea

Initial intent

Capture the feature, bug, user problem, or business goal in plain language.

2

Clarify

Scope decisions

Ask questions, identify edge cases, define non-goals, and remove ambiguity before writing tasks.

3

Requirements / PRD

Acceptance criteria

Convert the idea into visible behavior, constraints, success states, and testable outcomes.

4

Plan / Design

Technical path

Map files, APIs, data shape, UX states, risks, and the order of implementation.

5

Tasks

Executable slices

Split work into small steps that can be implemented, reviewed, tested, and committed.

6

Implementation

Code changes

Build the feature from the task list, ideally with focused tests and incremental commits.

7

Verify

Spec alignment

Run tests, inspect behavior, open the PR, and feed any mismatch back into the spec or tasks.

Core promise

The spec is the source of truth before code is written.

Feedback loop

Failed verification updates requirements, plan, or tasks instead of drifting silently.
